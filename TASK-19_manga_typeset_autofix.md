# TASK-19 (Phase 3, розширено): Автоматичне виправлення якості типсеттингу манги
### До того, як людина почне редагувати бульбашки

Це доповнення до `EDIT_METHODOLOGY_kindle-butch-gen.md`. Мета — по коду `translate_manga.py` знайдено конкретні причини трьох названих дефектів і запропоновано автоматичний QA/fix-прохід, який має відпрацювати **до** появи UI ручного редагування бульбашок (Phase 3 з попереднього документа).

---

## 1. Причини дефектів (знайдено в коді, не припущення)

### 1.1 "Подвійний текст" у бульбашці

Дві ймовірні причини, обидві реальні в поточному коді:

- **Детектор дає перекриті/дублікатні блоки.** `detector(img, ...)` повертає `blk_list` без дедуплікації. Якщо для однієї фізичної бульбашки детектор видав 2 перекриті bounding box'и (типовий збій для великих/овальних бульбашок), кожен обробляється **незалежно**: окремий OCR-кроп, окремий переклад, окремий typeset — у результаті на одному місці малюються два різні тексти одне поверх одного.
- **Слабкий inpaint залишає "привид" оригінального тексту.** `cv2.inpaint(img, mask_refined, 3, cv2.INPAINT_TELEA)` — радіус **3px**. Для жирного/великого шрифту оригіналу цього замало: залишаються напівпрозорі контури літер під новим текстом → візуально виглядає як накладення двох текстів.

### 1.2 Текст виходить за межі бульбашки

Корінь — мертвий фолбек у `fit_text()`:

```python
low, high = 8, 80
best_size = low           # 8
best_lines = [text]       # ВЕСЬ текст одним рядком, без переносу!
...
# якщо жоден розмір у бінарному пошуку не задовольнив умову — 
# цикл завершується, і повертається best_size/best_lines
# у їхньому ІНІЦІАЛІЗОВАНОМУ вигляді
```

Якщо перекладений (український) текст довший за бокс навіть при 8pt — а це типова ситуація, бо укр. переклад зазвичай на 20–40% довший за англ./яп. джерело, а бокс береться з розміру *оригінального* тексту — цикл жодного разу не проходить перевірку `total_height <= max_height and max_line_width <= max_width`. Функція повертає **не** розбитий на рядки текст при мінімальному шрифті, а **весь рядок цілим, без переносів**, розміром 8px. Це і є вихід за межі: суцільний довгий рядок ширший за бокс.

Додатково: `wrap_text()` переносить лише по пробілах — якщо одне слово довше за `max_width`, воно однаково потрапляє в рядок (`if width <= max_width or not current_line`), тобто рядок може перевищити ширину бокса навіть у "штатному" режимі.

### 1.3 Занадто малий / занадто великий шрифт

- **Занадто малий** — той самий мертвий фолбек (§1.2): при провалі підбору розмір падає до `low=8` без жодного нижнього порогу читабельності.
- **Занадто великий / непропорційний** — бокс для typeset береться з **тісного OCR bbox оригінального тексту** (`blk.xyxy`), а не з реальних меж бульбашки (контуру). Якщо оригінальний текст був коротким, а переклад — теж короткий, `fit_text` підбирає розмір, що ідеально заповнює тісний бокс, який сам по собі менший за фактичний внутрішній простір бульбашки → шрифт виглядає непропорційно великим відносно білого поля бульбашки.

## 2. Автоматичний QA/fix-прохід (виконується ДО ручного редагування)

П'ять окремих, незалежно тестованих стадій. Кожна — кандидат на окрему функцію в `translate_manga.py` (зараз усе це один монолітний `main()`).

| Стадія | Що робить | Виправляє |
|---|---|---|
| **A. Merge/dedup блоків** | IoU-злиття перекритих `blk_list` записів перед OCR (поріг, напр. IoU > 0.3 → один блок, union bbox) | подвійний текст (детектор) |
| **B. Надійна очистка** | Збільшити радіус inpaint (напр. 3→7) АБО двопрохідний inpaint, з фолбеком на суцільну заливку кольором фону бульбашки (медіана пікселів по контуру), якщо inpaint не впорався | подвійний текст (привид) |
| **C. Реальні межі бульбашки** | Використати контур маски (`mask_refined`), а не тісний OCR bbox, як typeset-бокс — з відступом всередину (padding), а не строго по краю тексту | overflow + непропорційний розмір |
| **D. Виправлений fit_text** | Прибрати мертвий фолбек; додати character-level hard-wrap для задовгих слів (переносити примусово); ввести нижню межу шрифту (мін. 12–14px) і верхню межу (макс. розумний % від висоти бокса) | занадто малий/великий шрифт, overflow |
| **E. Пост-рендер самоперевірка** | Після typeset — порахувати `overflow_ratio` (ширина рядка / ширина бокса) та `size_ratio` (обраний розмір / (min,max)); якщо межа порушена — записати `quality_flags.json` для сторінки/бульбашки | автоматично позначає залишкові випадки для черги ручного редагування (Phase 4 інфраструктура з основного документа) |

**Порядок важливий:** A → B → C → D → E. Кожна стадія зменшує кількість дефектів, які долетять до людини. Ціль — щоб ручне редагування бульбашок (raw Phase 3 з методики) залишалось для **словесних/перекладацьких** правок, а не для боротьби з рендер-багами.

## 3. Як це стикується з рештою TASK-19

- `PUT /api/edit/manga-text/<slug>/<page_filename>` і `POST /api/edit/regenerate-manga-page/<slug>/<page_filename>` (з основної методики) працюють **над результатом стадій A–E**, тобто людина вже бачить очищений, коректно типсетований варіант.
- `quality_flags.json` зі стадії E — природний вхід для вкладки "Pending Edits" (уже реалізованої для тексту/аудіо): сторінки/бульбашки з `overflow_ratio > поріг` чи `size_ratio` на межі мінімуму автоматично потрапляють у чергу на перегляд, навіть якщо ніхто їх вручну не позначив.
- `POST /api/edit/regenerate-manga-page` при регенерації однієї сторінки має прогонити її через **весь** новий пайплайн A–E, а не лише typeset — інакше проблема B (привид тексту) повернеться.

## 5. Тестування

У репозиторії вже є `test_manga.jpg` і `test_full_manga_ocr.py` — придатні як базовий regression-фікстур. Додати цільові кейси:
- сторінка з навмисно перекритими bbox (перевірка стадії A);
- бульбашка з довгим складним українським словом без пробілів (перевірка character-level wrap, стадія D);
- сторінка з жирним оригінальним шрифтом (перевірка стадії B на "привида").

Кожен фікстур перевіряється через `quality_flags.json` (стадія E) — після фіксу флагів для цих кейсів бути не повинно.
