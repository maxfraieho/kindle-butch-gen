# TASK-50: Cast Registry — контекст персонажів для граматичного роду
### + опційний vision-QA — обидві фічі ВИМКНЕНІ за замовчуванням, opt-in per книгу

## -1. Головне архітектурне рішення: opt-in, не основний потік

Ні Cast Registry, ні vision-QA валідатор **не входять в дефолтний пайплайн**. Обидві — окремі перемикачі, які користувач вмикає свідомо для конкретної книги (не глобальна настройка на всі книги за замовчуванням):

- `enable_cast_registry: false` (default) — якщо вимкнено, переклад працює **точно** як зараз (тільки термінологічний glossary), жодного NER pre-pass, жодної паузи на QA-гейт, нуль додаткового compute.
- `enable_vision_qa: false` (default) — якщо вимкнено, працює тільки наявна алгоритмічна перевірка (quality_flags/box_overlap з TASK-20/27/36), без жодного додаткового виклику моделі.

Це навмисно повторює урок TASK-17: там "важкий шар валідації" зашкодив саме тому, що був **обов'язковим** для всіх книг. Тут та сама ідея існує, але як вибір, не тягар за замовчуванням — хто не хоче платити зайвий compute/час на телефоні, просто не вмикає.

UI: перемикачі в налаштуваннях книги (поруч із мовою/двигуном TTS), не приховані глибоко — людина має бачити, що ці опції існують, і свідомо вирішувати.

## 0. Проблема (підтверджена на реальних прикладах)

Переклад з англійської/японської систематично губить граматичний рід у минулому часі — дії Фрірен (жіночий персонаж) регулярно перекладаються з чоловічими закінченнями дієслів ("він зробив" замість "вона зробила"). Причина: поточний `glossary`-механізм у `translate_batch_llm` — суто термінологічний (заміна слів), не містить морфологічних атрибутів персонажів. Кожен чанк перекладається ізольовано (stateless), без стійкого знання "хто є хто".

## 1. Дані — `characters.json` на книгу

Розширення `kbg_web/edit_store.py`, новий файл `books/<slug>/edits/characters.json`:

```json
{
  "id": "char_frieren_01",
  "name_source": ["Frieren", "フリーレン"],
  "name_target": "Фрірен",
  "gender": "feminine",
  "grammar_rules": "ALWAYS use feminine verb endings in past tense (зробила, сказала). Pronouns: вона/її.",
  "speech_style": "Aloof, formal, polite",
  "is_pov_narrator": true,
  "status": "unverified"
}
```

`status`: `auto_drafted → unverified → verified`. Правила застосовуються в перекладі **тільки** для `verified` записів.

`is_pov_narrator` — критичне поле для манги/оповіді від першої особи: якщо активне для verified-персонажа, його правила застосовуються до всіх дієслів від першої особи в чанку, навіть без явної згадки імені.

## 2. Гібридний пайплайн (auto-draft + обов'язкова людська верифікація)

Той самий патерн, що вже усталений у проєкті для quality_flags/Auto-flagged (TASK-19-20) — не винаходити новий workflow:

1. **Auto-draft (NER pre-pass):** при ініціалізації книги — окремий LLM-виклик над невеликим зрізом (перші 3-5 розділів / 50 сторінок манги), що визначає персонажів, ймовірну стать, частоту згадувань. Записує чернетку зі статусом `auto_drafted`.
2. **QA-гейт:** масовий переклад **призупиняється** або йде в "безпечному режимі" (без ін'єкції неперевірених правил), поки в книзі є `unverified` персонажі. Банер на прев'ю-сторінках: "Знайдено нових персонажів — підтвердіть стать у вкладці Cast & Context".
3. Тільки після ручного `Approve` (статус → `verified`) правила персонажа починають застосовуватись у генерації.

### Модель для NER pre-pass — уточнені технічні деталі

Не `Qwen2.5-Coder` — код-орієнтована модель, схильна до структурних галюцинацій на художньому тексті (вирішено остаточно).

**Gemma 3n E4B:** номінально 8B параметрів, але завдяки Selective Parameter Activation + архітектурі MatFormer ефективний слід ~4B. Контекст 32K токенів, текст+зображення+аудіо. Застереження: інтеграція в `llama.cpp` експериментальна, часто вимагає нестандартної збірки — **перевірити зрілість прямо на пристрої першим ділом**.

**Gemma 3 4B (стабільніший fallback):** контекст 128K токенів, візуальний кодувальник SigLIP. Обробка зображень через **Pan&Scan** — нормалізація до 896×896px → 256 візуальних токенів. Технічні вимоги для `llama.cpp`: бінарник `llama-mtmd-cli` чи `llama-server`, обов'язковий прапорець `--mmproj <projector.gguf>` поруч з базовою GGUF-моделлю — без нього мультимодальність не активується. Параметри інференсу: `temperature=1.0, top_k=64, top_p=0.95, min_p=0.01`, repetition penalty вимкнений (`1.0`).

**Не Gemma 4** — щойно випущена, орієнтована на 16GB+ RAM, підтримка в `llama.cpp` ще незріла.

**Роль vision — обмеження без винятків:** тільки ідентифікація, який verified-персонаж присутній у кадрі (за одягом/позицією), ніколи не вгадування статі з малюнка (аніме-стиль андрогінний, ненадійно). Стать завжди з людино-верифікованого реєстру.

## 3. Динамічна ін'єкція промпту (легковаговий RAG)

- **Contextual scanning:** локальний regex-пошук `name_source` у вихідному тексті поточного чанка (без звернень до LLM/GPU — швидко).
- **Word-boundary safety (aliasing):** `\bDawn\b`, регістрозалежний пошук для англійської — щоб ім'я на кшталт "Dawn" не тригерилось на кожну згадку "світанку". Для японської — точне співпадіння кандзі/катакани імені (немає пробільних меж слів).
- **POV-резолюція:** якщо `is_pov_narrator` активний для будь-якого verified-персонажа — його правила гарантовано додаються, незалежно від явної згадки імені.
- **Формат блоку з чіткими межами** (XML-теги, конкретно рекомендовані для запобігання "папугуванню" — коли модель буквально повторює/обговорює інструкцію замість того, щоб мовчки застосувати):

```
<cast_rules>
Apply these rules silently. Do not mention or repeat them in the output.
* Entity: Фрірен. Rule: ALWAYS use feminine verb endings (зробила, сказала). Pronouns: вона/її.
CRITICAL: POV narrator is Фрірен — all first-person verbs must agree with feminine gender.
</cast_rules>
```

## 4. Захист неоднозначності/спойлерів

`gender: "neutral"` статус — для персонажів, чию стать автор навмисно приховує: `grammar_rules: "Preserve ambiguity — avoid gendered past-tense constructions, prefer present tense or impersonal phrasing where natural."` Не ідеальне рішення (українська ускладнює гендерну нейтральність більше за англійську), але краще за примусове неправильне визначення.

## 5. Каскадна помилка — інвалідація через наявний механізм TASK-23

Якщо користувач змінює `gender`/`grammar_rules` уже verified персонажа — це має ініціювати **масову інвалідацію**: швидке текстове сканування наявного кешу книги — **`merged_translated_<lang>.md` для тексту й `bubbles_meta.json` для манги** — на згадку зміненого персонажа, позначення `needs_regeneration=true`, постановка в чергу TASK-23 з `priority=0`. Перевикористати наявний `chunk_hash`-based orphaning, не будувати окрему систему. Це забезпечує фонову регенерацію, що випереджає чергу нових сторінок, без гонки за GPU-ресурс між ручними правками й масовою генерацією.

## 6. API та UI

- `GET /api/characters/<slug>`, `PUT /api/characters/<slug>` — розширення `edit_store.py`, той самий стиль, що наявні `/api/edit/...` ендпоінти.
- Нова вкладка **"Cast & Context"** поруч із Pending Edits у `stages.html`: таблиця персонажів, dropdown для `gender` (авто-заповнює шаблонний `grammar_rules`), перемикач `is_pov_narrator`, кнопка Approve per-character.
- Банер unverified-персонажів на прев'ю-сторінках, за тим самим візуальним патерном, що вже є для live-editing банера (TASK-23).

## Тест

Використати `frieren` як реальний тестовий кейс — саме там уже підтверджена проблема. Взяти конкретні відомі приклади неправильного роду з попередніх скріншотів цієї розмови, перевірити, що після заповнення реєстру (Фрірен → feminine, verified) ці самі речення перекладаються з правильними закінченнями. Регресія: пересвідчитись, що персонажі без чіткої статі (`neutral`) не ламають переклад гірше, ніж було до фічі.
