## [x] TASK-18: [BUG][FIXED] Race condition in /api/models/start allows duplicate llama-server processes
* **Problem:** `/api/models/start` kills any existing `llama-server` on port 8081 (`pkill -f "llama-server.*8081"`) then launches `~/start-translation-server.sh`, which *itself* does its own pkill + 2s sleep before launching. Two Start requests fired close together (e.g. an impatient double-tap on the "Start Server" button) race past both pkill checkpoints and each spawns its own server. Reproduced empirically: firing two `/api/models/start` calls 0.3s apart left **two full 4.4GB Hy-MT2-7B llama-server processes running simultaneously** (confirmed via `ps aux` on the phone), and the state did not self-correct over a 60s observation window ŌĆö required a manual `kill -9` to clean up. On a memory-constrained phone this is a real resource-exhaustion risk.
* **Solution:**
  1. Replaced `pkill -f "llama-server.*8081"` pattern-matching with a PID file (`~/llama-server-8081.pid`) as the single source of truth for the tracked process ŌĆö `_read_llama_pid()` / `_stop_llama_server()` in `app.py` verify liveness via `os.kill(pid, 0)` and always clear the file afterward so a stale entry never lingers.
  2. Added an `O_CREAT|O_EXCL` start lock (`~/llama-server-8081.lock`) around the stop-old+launch-new critical section in `/api/models/start`. ## [x] TASK-36 follow-up: [BUG][FIXED] resolved manga edits didn't survive a LATER, unrelated regeneration of the same page
* **Problem, prioritized correctly over TASK-33 per explicit instruction:** the "loose end" flagged above turned out to be the most important finding in the whole TASK-36 report, not a footnote - `process_page()` re-runs a page's OCR+LLM translation completely from scratch on every regeneration, so an edit that had already reached `status="regenerated"` in a PAST run was invisible to app.py's/translate_manga.py's override-building code, which only ever looked at `status="pending"` edits. TASK-36 itself made this bug's real-world blast radius much larger: **manual geometry editing on any single bubble now regenerates the ENTIRE page**, so every future geometry-only fix on a page that has an older, already-approved text fix would silently roll that text back to a fresh (and possibly different) raw translation - invisible until the page is reopened. Exactly the "quietly eats already-done work" class of bug.
* **Fix (`kbg_web/app.py`'s `edit_regenerate_manga_page`, `translate_manga.py`'s `apply_pending_manga_edits`):** the "should a regen happen at all" gate is unchanged - still requires at least one genuinely new `status="pending"` edit for the page (so clicking Regenerate on an already-fully-applied page is still a no-op). After a successful regen, ALL included edits (old and new alike) get their status re-confirmed via the same IoU IDreconciliation as before - `"regenerated"` (fresh `applied_at`) if their bubble still resolves, `"orphaned"` if not.
* **Deliberately out of scope, disclosed rather than silently expanded:** the separate `"approved"` status (reachable directly from `"pending"` via the Approve button in the Pending Edits tab, which for manga mode does **not** touch the rendered image at all - existing, pre-TASK-36 behavior, confirmed by re-reading `edit_approve`'s own code comment) was deliberately **not** added to the reapply set. ### Problem statement and motivation
`kbg_web/edit_store.py` backs every non-destructive edit (manga text/geometry/font-size, regular-book text, stress-marking, audio) with a single flat JSON file per book (`<book_dir>/edits/edits.json`), read via `_load_edits()` (full-file `json.load`) and written via `_save_edits()` (full-file `json.dump`, no atomic rename, no locking) on every `add_edit()`/`mark_status()` call. A read-modify-write cycle with no locking between two writers is a textbook lost-update race: if the background process and a live UI edit both call `add_edit()`/`mark_status()` close enough in time, one write can silently clobber the other, without any error surfacing to either side.

### Architectural scope and affected components
**Must change:** `kbg_web/edit_store.py` itself (`_load_edits`, `_save_edits`, `add_edit`, `list_edits`, `get_edit`, `mark_status`, `_edits_path`) - the entire module's storage backend.

**Direct call sites (grepped precisely, not the wider GitNexus fan-out through the shared `resolve_book_paths` helper - see Dependency analysis below for why that number is misleading):**

### Dependency analysis
`edit_store.py`'s current public API (`add_edit`, `list_edits`, `get_edit`, `mark_status`) is small and already well-factored as a module boundary - **the migration should be designed as a drop-in replacement preserving these exact function signatures**, so none of the ~9 caller functions above need to change at all, only `edit_store.py`'s internals. This is the single most important design constraint for keeping this migration's actual blast radius small despite the large number of indirect callers - if the public API doesn't change shape, the 20-process GitNexus number becomes irrelevant (those callers depend on `edit_store`'s behavior, not its storage format).

### Technical risks and mitigations
1. **Data migration correctness (irreversible if botched)** - existing `edits.json` files for real, in-production books (`frieren`, `vibe-programming`, `three-days-of-happiness`, etc.) must convert to the new schema with zero data loss. *Mitigation*: write a migration script, keep the original `edits.json` as a `.bak` untouched, verify row-count and spot-check content equality before ever deleting/renaming the original.
2. **Concurrency semantics not actually validated in THIS environment** - the research report endorses SQLite+WAL in general, but this project runs inside `proot-distro` (a ptrace-based syscall interception layer, not a true container), and SQLite's locking primitives ultimately rely on `fcntl`/POSIX advisory locks, which is exactly the class of primitive the report separately warns is unreliable under Android's I/O scheduler. **Do not assume the general recommendation transfers correctly to this specific runtime without direct verification.** *Mitigation*: as the very first implementation step, before any schema/design work, run an isolated experiment - two separate Python processes inside the same `proot-distro` container, one looping writes with `WAL` mode enabled, one reading concurrently, verified for corruption/lost-write over a sustained run - before committing to this approach at all.
3. **Multiple-writer assumption may not hold** - WAL mode cleanly supports one writer + many readers; verify the actual concurrency pattern is genuinely single-writer-at-a-time in practice (background pipeline OR live UI edit, not both writing simultaneously in normal operation) rather than assuming it.
4. **Schema design lock-in** - once real data lives in SQLite, changing the schema is a migration itself. *Mitigation*: design the schema to mirror the current JSON record shape closely (`id, slug, mode, target_id, field, original_value, edited_value, status, created_at, applied_at`) rather than over-normalizing, keeping the mapping from old to new format trivial to reason about and re-verify later if needed.

### Recommended implementation order
1. Empirical validation of SQLite+WAL concurrent read/write inside the actual `proot-distro` Ubuntu environment (isolated experiment, zero production impact).
2. Design the schema + rewrite `edit_store.py`'s internals with the *same* public function signatures - a true drop-in replacement.
3. Write and test a migration script (JSON ŌåÆ SQLite) against a **copy** of real book data, not live production data.
4. Migrate one low-stakes/test book first, verify end-to-end through the actual UI (create edit, list, approve/discard, regenerate).
5. Migrate all real books only once step 4 confirms correctness, retaining `.bak` originals.
6. Observe a real background-generation-plus-live-edit session (TASK-23's own scenario) for any residual race-condition symptoms.

### Milestones and acceptance criteria
- **M1:** Concurrent WAL read/write proven corruption-free inside the real `proot-distro` container (empirical, not assumed).
- **M2:** New `edit_store.py` passes identical behavior against the old one for every existing function signature, using a fresh SQLite DB (no callers changed).
- **M3:** Migration script converts a real `edits.json` with 100% row-count/content match, verified against the `.bak`.
- **M4:** A live end-to-end test (create edit via UI ŌåÆ regenerate ŌåÆ status transitions) behaves identically to the current JSON-backed system.
- **M5:** No regression across every edit mode sharing this module (manga text/bbox, regular-book text, stress, audio) - this is shared infrastructure, so all modes need re-verification, not just manga.

## [x] TASK-40: [BUG][FIXED] Timed-out single-page regen left the actual worker process orphaned, only killed its proot wrapper
* **Problem, flagged as a loose end during TASK-36 testing, closed out separately** (isolated, low-risk, doesn't touch `edit_store`/`process_page` - explicitly kept apart from the deferred TASK-38/39 refactors): `kbg_web/app.py`'s `edit_regenerate_manga_page()` ran the regen subprocess via `subprocess.run(cmd, timeout=180)`. When the 180s timeout fired, Python's default `TimeoutExpired` handling kills only the *direct* child - the `proot-distro` wrapper process (`cmd[0]`) - not any of its own descendants. Since `cmd` is `proot ... /bin/bash -c "python3 translate_manga.py ..."`, killing just the wrapper left the actual `python3` worker running *inside* the PRoot container as an orphan, unmonitored, for several more minutes past the point Flask (and the client) had already given up and returned an error. On `TimeoutExpired`, `os.killpg(os.getpgid(proc.pid), signal.SIGKILL)` kills the **entire** process group in one call, not just the wrapper, then re-`communicate()`s once more to reap the killed process and avoid a zombie. Downstream code (which reads `res.returncode`/`res.stdout`/`res.stderr`) needed zero changes - the new code constructs a `subprocess.CompletedProcess` from the `Popen` results, keeping the diff scoped to just the process-spawning block.
* **Verification:** (1) local mechanism proof - a synthetic nested `bash -c "sleep 100 & wait"` process tree, run once the old way (`.kill()` on the direct child only) and once the new way (`start_new_session=True` + `os.killpg`): the old approach measurably left the grandchild `sleep` process running after the "timeout", the new approach's `killpg` left nothing behind. (2) Real-device happy-path confirmation: deployed to production, triggered a real regen through the actual `/api/edit/regenerate-manga-page` Flask route (not a CLI bypass) with a font-size+geometry edit, confirmed the edit correctly reached `status="regenerated"` and the resulting typeset PNG visually shows the requested larger font - proves the refactor didn't break the normal (non-timeout) success path. The true timeout path wasn't forced live against production code (would require an artificial multi-minute hang injected into a live pipeline run, judged not worth the risk for this fix) - covered instead by the local mechanism proof, which isolates and directly demonstrates the actual kill behavior this fix changes.
