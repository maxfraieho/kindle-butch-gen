## [x] TASK-18: [BUG][FIXED] Race condition in /api/models/start allows duplicate llama-server processes
* **Problem:** `/api/models/start` kills any existing `llama-server` on port 8081 (`pkill -f "llama-server.*8081"`) then launches `~/start-translation-server.sh`, which *itself* does its own pkill + 2s sleep before launching. Two Start requests fired close together (e.g. an impatient double-tap on the "Start Server" button) race past both pkill checkpoints and each spawns its own server. Reproduced empirically: firing two `/api/models/start` calls 0.3s apart left **two full 4.4GB Hy-MT2-7B llama-server processes running simultaneously** (confirmed via `ps aux` on the phone), and the state did not self-correct over a 60s observation window — required a manual `kill -9` to clean up. On a memory-constrained phone this is a real resource-exhaustion risk.
* **Solution:**
  1. Replaced `pkill -f "llama-server.*8081"` pattern-matching with a PID file (`~/llama-server-8081.pid`) as the single source of truth for the tracked process — `_read_llama_pid()` / `_stop_llama_server()` in `app.py` verify liveness via `os.kill(pid, 0)` and always clear the file afterward so a stale entry never lingers.
  2. Added an `O_CREAT|O_EXCL` start lock (`~/llama-server-8081.lock`) around the stop-old+launch-new critical section in `/api/models/start`. A concurrent start request now gets an explicit `409 "A start is already in progress"` instead of silently racing through. The lock self-clears if older than 15s (far longer than the section normally takes), so a crashed request can't permanently block future starts.
  3. Removed the duplicate `pkill`+`sleep 2` from `~/start-translation-server.sh` (phone-only, not tracked in git) — stopping is now exclusively the API layer's responsibility; the script only launches and writes its own PID to the file path passed in as `$1`.
  4. `dashboard.html`'s Start Server button now stays disabled ("Starting...") for up to 90s, polling `/api/models` until it reports `running: true`, instead of re-enabling right after the fire-and-forget POST returns — closes the double-click window client-side too.
* **Verification Method:** All via `curl`+SSH against the live phone (agent-workspace/AGY3 still unreachable). (a) Fired two `/api/models/start` 0.2s apart → first got `200`, second got `409 "A start is already in progress"`; after model load, exactly 1 `llama-server` process and the PID file matched it. (b) Normal Stop→Start: confirmed PID file removed on stop (and process actually gone via `ps aux`, not just a UI illusion), then correctly re-created with the new PID on start. (c) Simulated a stale lock (manually aged to 20s) and confirmed the next start self-healed past it (`200`, not `409`) instead of being permanently blocked. Device restored to normal running state afterward. Full report: ai-memory `notes/2026-07-15-models-manager-test-report.md` and follow-up fix notes (project `kindle-butch-gen`).
* **Type:** `direct`

### Problem statement and motivation
`kbg_web/edit_store.py` backs every non-destructive edit (manga text/geometry/font-size, regular-book text, stress-marking, audio) with a single flat JSON file per book (`<book_dir>/edits/edits.json`), read via `_load_edits()` (full-file `json.load`) and written via `_save_edits()` (full-file `json.dump`, no atomic rename, no locking) on every `add_edit()`/`mark_status()` call. Two independent processes can legitimately touch this file concurrently on a real device: the Flask web UI (`kbg_web/app.py`, live human edits) and the background `translate_manga.py` process running inside `proot-distro` (`apply_pending_manga_edits()`, invoked between pages while `main()`'s full-book loop is still running - this is TASK-23's own live-editing feature, so this isn't a hypothetical, it's the system's actual designed concurrency model). A read-modify-write cycle with no locking between two writers is a textbook lost-update race: if the background process and a live UI edit both call `add_edit()`/`mark_status()` close enough in time, one write can silently clobber the other, without any error surfacing to either side.

The research report reviewed this session (`Мобільні AI-пайплайни для перекладу.txt`, section 3, "'Живе' редагування та конфлікти ресурсів") names this exact pattern - plain file locks in a Termux/Android I/O-scheduler environment - as unreliable, and recommends SQLite with `PRAGMA journal_mode=WAL` as the standard, proven pattern for a single background Writer + concurrent UI Reader/Writer on a resource-constrained mobile device, plus a trivial `priority` column for a live-edit-jumps-the-queue pattern the current architecture has no equivalent of at all (a live edit currently just waits for whatever `apply_pending_manga_edits()`'s next page-boundary check happens to be, per TASK-23's existing design - no priority signal exists to make it happen sooner).

### Architectural scope and affected components
**Must change:** `kbg_web/edit_store.py` itself (`_load_edits`, `_save_edits`, `add_edit`, `list_edits`, `get_edit`, `mark_status`, `_edits_path`) - the entire module's storage backend.

**Direct call sites (grepped precisely, not the wider GitNexus fan-out through the shared `resolve_book_paths` helper - see Dependency analysis below for why that number is misleading):**
- `translate_manga.py`: `apply_pending_manga_edits()` - 2 call sites (`list_edits`, `mark_status`).
- `run_conversion_batches.py`: `main()` - 2 call sites (`list_edits`, `mark_status`), text-edit mode for regular books.
- `kbg_web/app.py` - ~13 call sites across these route handlers: `edit_text` (regular-book text edits), `edit_stress` (stress-marking), `edit_queue` (GET list, used by every Pending-Edits UI fetch), `edit_regenerate_audio`, `edit_approve` (shared across all edit modes), `edit_discard` (shared across all edit modes), `edit_manga_text`, `edit_manga_bbox` (TASK-36), `edit_regenerate_manga_page`.

**Not required to change:** `common/book_paths.py`'s `resolve_book_paths` (GitNexus's impact query traced through this shared helper to inflate the affected-process count to 20 - those are mostly unrelated routes like `add_book_api`/`delete_book_api`/`upload_file_api` that call `resolve_book_paths` for entirely different reasons and never touch `edit_store` at all; the true dependency surface is the ~9 functions listed above).

### Dependency analysis
`edit_store.py`'s current public API (`add_edit`, `list_edits`, `get_edit`, `mark_status`) is small and already well-factored as a module boundary - **the migration should be designed as a drop-in replacement preserving these exact function signatures**, so none of the ~9 caller functions above need to change at all, only `edit_store.py`'s internals. This is the single most important design constraint for keeping this migration's actual blast radius small despite the large number of indirect callers - if the public API doesn't change shape, the 20-process GitNexus number becomes irrelevant (those callers depend on `edit_store`'s behavior, not its storage format).

### Technical risks and mitigations
1. **Data migration correctness (irreversible if botched)** - existing `edits.json` files for real, in-production books (`frieren`, `vibe-programming`, `three-days-of-happiness`, etc.) must convert to the new schema with zero data loss. *Mitigation*: write a migration script, keep the original `edits.json` as a `.bak` untouched, verify row-count and spot-check content equality before ever deleting/renaming the original.
2. **Concurrency semantics not actually validated in THIS environment** - the research report endorses SQLite+WAL in general, but this project runs inside `proot-distro` (a ptrace-based syscall interception layer, not a true container), and SQLite's locking primitives ultimately rely on `fcntl`/POSIX advisory locks, which is exactly the class of primitive the report separately warns is unreliable under Android's I/O scheduler. **Do not assume the general recommendation transfers correctly to this specific runtime without direct verification.** *Mitigation*: as the very first implementation step, before any schema/design work, run an isolated experiment - two separate Python processes inside the same `proot-distro` container, one looping writes with `WAL` mode enabled, one reading concurrently, verified for corruption/lost-write over a sustained run - before committing to this approach at all.
3. **Multiple-writer assumption may not hold** - WAL mode cleanly supports one writer + many readers; verify the actual concurrency pattern is genuinely single-writer-at-a-time in practice (background pipeline OR live UI edit, not both writing simultaneously in normal operation) rather than assuming it.
4. **Schema design lock-in** - once real data lives in SQLite, changing the schema is a migration itself. *Mitigation*: design the schema to mirror the current JSON record shape closely (`id, slug, mode, target_id, field, original_value, edited_value, status, created_at, applied_at`) rather than over-normalizing, keeping the mapping from old to new format trivial to reason about and re-verify later if needed.

### Recommended implementation order
1. Empirical validation of SQLite+WAL concurrent read/write inside the actual `proot-distro` Ubuntu environment (isolated experiment, zero production impact).
2. Design the schema + rewrite `edit_store.py`'s internals with the *same* public function signatures - a true drop-in replacement.
3. Write and test a migration script (JSON → SQLite) against a **copy** of real book data, not live production data.
4. Migrate one low-stakes/test book first, verify end-to-end through the actual UI (create edit, list, approve/discard, regenerate).
5. Migrate all real books only once step 4 confirms correctness, retaining `.bak` originals.
6. Observe a real background-generation-plus-live-edit session (TASK-23's own scenario) for any residual race-condition symptoms.

### Milestones and acceptance criteria
- **M1:** Concurrent WAL read/write proven corruption-free inside the real `proot-distro` container (empirical, not assumed).
- **M2:** New `edit_store.py` passes identical behavior against the old one for every existing function signature, using a fresh SQLite DB (no callers changed).
- **M3:** Migration script converts a real `edits.json` with 100% row-count/content match, verified against the `.bak`.
- **M4:** A live end-to-end test (create edit via UI → regenerate → status transitions) behaves identically to the current JSON-backed system.
- **M5:** No regression across every edit mode sharing this module (manga text/bbox, regular-book text, stress, audio) - this is shared infrastructure, so all modes need re-verification, not just manga.

## [ ] TASK-39: [PLANNED, DEFERRED] Granular stage invalidation for process_page (avoid full OCR+LLM re-run on geometry/font-only edits)
* **Status:** planning only - **no code changes made**. Deferred deliberately: this is a larger, higher-design-risk undertaking than TASK-38 (see below - it requires changing the persisted artifact schema most of the rest of the system depends on, not just an internal storage swap), and depends on open architectural questions (cross-bubble padding coupling, see Risk 2) that need dedicated thinking time before implementation should start at all.

### Problem statement and motivation
`process_page()` re-runs the entire Stage A-E pipeline (detect → dedupe → OCR → translate → inpaint → typeset → post-check) from scratch on every single regeneration, even when a human edit only changes a bubble's box position or font size and touches neither the source image nor the translated text. This is the root architectural cause behind the bug fixed this session as "TASK-36 follow-up" (a `status="regenerated"` text edit not surviving a *later*, unrelated geometry-only regeneration of the same page) - that fix (widening the override set to reapply already-"regenerated" edits on every future regen) works and is deployed, but it's a symptom-level patch: the page's OCR and LLM translation calls still re-run in full every time, which is wasteful of the two most expensive resources on this device (network/LLM latency and CPU-bound OCR) for a purely cosmetic change. TASK-36 also made this dramatically more likely to matter in practice, since geometry editing on a single bubble now regenerates the whole page routinely, not just on rare full re-translations.

### Architectural scope and affected components
**Core function to restructure:** `translate_manga.py:process_page()` - 13 direct dependents per GitNexus (`wrap_text`, `fit_text`, `draw_text_centered`, `real_word_fraction`, `clean_ocr_edge_noise`, `dedupe_blocks`, `robust_inpaint`, `_get_bubble_core_box`, `_apply_padding_neighbor_safe`, `post_render_check`, `compute_box_overlap_flags`, `assign_bubble_ids`, `translate_batch_llm`), 22 total impacted symbols.

**All three of `process_page`'s own callers need updating**, not just the function itself: `regenerate_single_page()` (CLI/subprocess path), `apply_pending_manga_edits()` (in-process live-regen path, TASK-23), and `main()`'s own full-book per-page loop.

**This is fundamentally an artifact-schema change, not just a code refactor** - "granular invalidation" only works if each expensive stage's output (detected boxes, raw OCR text, translations) is itself persisted somewhere a *later, separate* invocation can read back, rather than living only in the Python call stack of one `process_page()` call as it does today. Currently only the *final* `bubbles_meta.json` is persisted, which conflates every stage's output into one record with no way to tell "this text came from OCR run X" apart from "this box came from detection run Y." This makes the scope meaningfully larger than TASK-38: it isn't just swapping a backend behind a stable API, it's introducing a new intermediate artifact that other things (the frontend overlay, `backfill_bubbles_meta`, quality-flag computation, CBZ export) don't currently know about and must not be broken by.

### Dependency analysis
Recommended design constraint to keep this contained: add a **new, additive, page-scoped cache file** (e.g., `<book_dir>/pipeline_cache/<page_stem>.json`, holding detected boxes + raw OCR text + translations) that sits *alongside* `bubbles_meta.json` rather than replacing or restructuring it - this means the existing schema every other component already depends on (frontend overlay rendering, TASK-25's `backfill_bubbles_meta`, TASK-36's `compute_box_overlap_flags`, CBZ packaging) needs zero changes. Only `process_page()`'s own internals and its three callers need to know the cache exists.

### Technical risks and mitigations
1. **Cross-bubble coupling makes "stage" boundaries less clean than they sound.** TASK-36's neighbor-aware padding (`_apply_padding_neighbor_safe`) means every bubble's box on a page depends on every *other* bubble's box on the same page - "only re-run typesetting for the edited bubble" isn't actually correct in isolation; the padding/box-overlap computation is inherently page-wide, not per-bubble. *Mitigation*: the fast path must still recompute Stage C (box padding) and the overlap/quality-flag pass for the **whole page**, using cached OCR+translation - it's specifically Stage OCR and Stage Translate (the genuinely expensive, per-bubble-content-dependent stages) that get skipped, not the page-wide geometry pass.
2. **Determining safe invalidation triggers.** The cache must be treated as stale (fall back to a full run) whenever: a *text* edit exists for the page (translation changed, cache's translation is wrong), the source image changed, or the cache is simply absent/corrupt/version-mismatched. Conservative-by-default: when in doubt, do a full run rather than trust a possibly-stale cache.
3. **Detection instability across runs.** `dedupe_blocks`'s bubble count/order isn't guaranteed identical between two independent detection runs (this is *why* `match_bubbles_iou` reconciliation exists at all today) - if detection is cached and skipped, that's actually a *feature* here (avoids the reconciliation problem entirely for the fast path), but only as long as the fast path is never taken when the source image genuinely changed.
4. **Bigger payoff, bigger risk, than TASK-38** - the actual benefit (compute/battery savings) should be measured empirically before deciding this is worth the added persistent-format complexity; if TASK-38's SQLite migration alone, or the already-deployed TASK-36-follow-up band-aid, prove sufficient in practice, this task may not be worth the added complexity at all. Recommend re-evaluating necessity after TASK-38 ships and has been observed in real use for a while.

### Recommended implementation order
1. Design the pipeline-cache file format (additive, zero changes to `bubbles_meta.json`'s existing schema) - review and sign off on the design before writing code.
2. Implement cache *writing* only first (persist Stage A/OCR/Translate outputs on every existing regen call, consumed by nothing yet) - a safe, purely observable first step with no behavior change.
3. Implement the "read cache, skip straight to page-wide box/typeset/flags" fast path, gated strictly to geometry/font-only edits, with conservative fallback to full regen for anything else.
4. Test with the same real-device, visual-before/after rigor this session established for TASK-33/36: confirm the fast path produces byte-identical OCR/translation output to what a full regen would have produced, on multiple real multi-bubble pages (not just the single-bubble pages this session happened to test with, given Risk 1's cross-bubble coupling).
5. Only after the fast path is proven reliable, measure real compute/battery savings on-device before deciding whether to keep, extend, or remove the TASK-36-follow-up band-aid (defense-in-depth is fine either way, but the measurement should inform the call).

### Milestones and acceptance criteria
- **M1:** Pipeline-cache format designed and reviewed (not yet consumed by anything).
- **M2:** Cache correctly written on every existing regen call, verified content-correct, zero behavior change to current output.
- **M3:** Fast path implemented for geometry/font-only edits; produces byte-identical OCR/translated text to a full regen on multiple real multi-bubble pages.
- **M4:** Real on-device timing/battery comparison (fast path vs. full regen) demonstrates the savings justify the added complexity - if not, document that finding and reconsider whether to proceed further.
- **M5:** No regression in TASK-36's neighbor-aware padding/overlap-detection logic when the fast path runs on a page with multiple bubbles, only one of which was actually edited.

## [x] TASK-40: [BUG][FIXED] Timed-out single-page regen left the actual worker process orphaned, only killed its proot wrapper
* **Problem, flagged as a loose end during TASK-36 testing, closed out separately** (isolated, low-risk, doesn't touch `edit_store`/`process_page` - explicitly kept apart from the deferred TASK-38/39 refactors): `kbg_web/app.py`'s `edit_regenerate_manga_page()` ran the regen subprocess via `subprocess.run(cmd, timeout=180)`. When the 180s timeout fired, Python's default `TimeoutExpired` handling kills only the *direct* child - the `proot-distro` wrapper process (`cmd[0]`) - not any of its own descendants. Since `cmd` is `proot ... /bin/bash -c "python3 translate_manga.py ..."`, killing just the wrapper left the actual `python3` worker running *inside* the PRoot container as an orphan, unmonitored, for several more minutes past the point Flask (and the client) had already given up and returned an error. Observed live twice during TASK-36 testing; the orphan happened to complete correctly on its own both times, but that was luck, not a guarantee - a genuinely hung run would have run forever, silently burning battery/CPU with nothing watching it.
* **Fix:** replaced the single `subprocess.run(...)` call with `subprocess.Popen(cmd, ..., start_new_session=True)` (puts the whole `proot → bash → python3` chain in its own process group) + explicit `proc.communicate(timeout=180)`. On `TimeoutExpired`, `os.killpg(os.getpgid(proc.pid), signal.SIGKILL)` kills the **entire** process group in one call, not just the wrapper, then re-`communicate()`s once more to reap the killed process and avoid a zombie. Downstream code (which reads `res.returncode`/`res.stdout`/`res.stderr`) needed zero changes - the new code constructs a `subprocess.CompletedProcess` from the `Popen` results, keeping the diff scoped to just the process-spawning block.
* **Verification:** (1) local mechanism proof - a synthetic nested `bash -c "sleep 100 & wait"` process tree, run once the old way (`.kill()` on the direct child only) and once the new way (`start_new_session=True` + `os.killpg`): the old approach measurably left the grandchild `sleep` process running after the "timeout", the new approach's `killpg` left nothing behind. (2) Real-device happy-path confirmation: deployed to production, triggered a real regen through the actual `/api/edit/regenerate-manga-page` Flask route (not a CLI bypass) with a font-size+geometry edit, confirmed the edit correctly reached `status="regenerated"` and the resulting typeset PNG visually shows the requested larger font - proves the refactor didn't break the normal (non-timeout) success path. The true timeout path wasn't forced live against production code (would require an artificial multi-minute hang injected into a live pipeline run, judged not worth the risk for this fix) - covered instead by the local mechanism proof, which isolates and directly demonstrates the actual kill behavior this fix changes.
* **Unrelated observation made during this verification, not investigated further:** the device showed increasingly tight memory (343MB free of 15GB, ~3.7GB swap in use) during this session's later regen tests, plausibly explaining some of the longer-than-expected regen times observed throughout TASK-33/36 testing. Independent of this fix; noted for awareness, not chased.
* **Type:** `direct` - isolated, self-contained, low architectural risk, explicitly separate from the deferred TASK-38/39 work, fixed and verified both locally (mechanism) and live (happy path, real device).

## [x] TASK-45: [BUG][FIXED] Auto-resumed conversion never cleared .active_conversion.json — every later Termux restart relaunched the pipeline
* **Problem (found live, not hypothetical):** after the completed TASK-42 frieren run, a stale `.active_conversion.json` still sat on the device with no conversion process running. Root cause: `bin/resume_active_conversion.py` deliberately kept the state file so a re-interrupted resume could resume again — but NOTHING ever cleared it when a resumed run *completed*, because Flask's `handle_process_completion` only fires for processes Flask itself started. Consequences: every subsequent Termux restart relaunched the pipeline against a finished book (wake-lock, battery, CBZ rebuild), and a genuinely FAILING resumed run would be retried forever on every boot — the exact infinite-retry hazard app.py's TASK-41 design deliberately avoided on its own path.
* **Fix (commit `e8831ef`):** the resume script now stays alive as a watcher: `proc.wait()`, then clears the state file on ANY exit (success or failure), mirroring `handle_process_completion` semantics. If the environment dies mid-run the watcher dies with it and the file correctly survives for the next boot. Guarded against deleting a NEWER state file Flask wrote for a different run in the meantime (compares saved `cmd`).
* **Verification (live on device, real code path):** deployed via the new TASK-44 `git pull` path; synthetic state file (`sh -c "sleep 2; echo resumed-run-finished"`) → ran `bin/resume_active_conversion.py` → it launched the run, waited, reported `exited (code 0); cleared ...`, state file confirmed gone, log contains the marker line.
